# Codex 任务包

这里放的是给 Codex(或任何外部 agent)的可执行任务规格。每个任务一个 `.md`,包含:

- **背景与动机**:为什么做、依据哪条审查发现
- **作用域**:产出哪些文件、目录、新 API
- **契约**:函数签名 + 输入输出 schema + 关键不变量
- **测试**:必须新增的测试用例(语义、边界、回归)
- **验收**:跑哪些命令必须全绿才算完成
- **反需求**:明确不能做的事(避免越界、避免破坏其他模块)

## 给 Codex 的通用工程约定

读这一节,所有任务都适用。**不要**在每个任务里自己再发明一套约定。

### 仓库与分支

- 项目根: `/Users/wenruifeng/Documents/github/stock`(本机) / `home-debian:/home/wenruifeng/projects/stock`(生产)
- 主分支: `main`,远端 `origin`
- 每个任务起一个分支: `feat/<short-slug>`(新功能) 或 `fix/<short-slug>`(修缺陷)
- **绝不**直接在 `main` 上工作
- 一个任务一个 PR(commit 可以多个,但应主题内聚)

### Python / 测试

- Python 3.13,虚拟环境 `.venv`
- 测试用 `pytest`。**纯单元测试不连数据库**(用合成 DataFrame / fixture)。需要 PostgreSQL 的集成测试加 `pytest.mark.integration` 装饰,会自动用 conftest 的 `pg_db` fixture(临时集群,完事销毁)
- 跑测试:
  - 纯单元: `python -m pytest tests/ -q -m "not integration"`
  - 全部: `python -m pytest tests/ -q`
- **新代码必须配测试**,核心语义都要有断言锁
- 不引入新依赖,坚持 `pandas / numpy / sqlalchemy`。需要新包必须先在 PR 描述里说明并改 `requirements.txt`

### 风格

- **不写多行 docstring 也不写无意义注释**。doc 短句即可,只写"why"不写"what"。命名好的代码自带 what。
- **不创建 README 或 docs 文件**,除非任务明确要求
- 中文 docstring 是项目惯例(看 `research/fundamentals.py` 风格),沿用
- emoji 除非明示请求,**不要用**
- 错误处理:不要捕异常然后忽略;不要在内部代码做无用 validation。trust 内部调用,只在系统边界(用户输入、外部 API)校验

### 数据/Schema 不变量

读 `CLAUDE.md` 顶部的"核心架构规则"。简版:

1. **`security_id` 是持久身份**,`symbol` 是可变属性,绝不用 symbol 做 join 键
2. **`daily_prices` 只存 raw OHLCV**,不存复权价、不存技术指标
3. **复权因子只在 `vendor_adjustment_factors` / `computed_adjustment_factors` 两张层**
4. **`historical_shares.filing_date` 是 PIT 边界**,任何用 shares 的 PIT 面板必须以它为可见性边界
5. **`sec_fundamental_facts.filed_date`(+1 天默认 visible_delay)是基本面 PIT 边界**
6. **不存派生比率回事实表**。read-time 算。

### 研究层约定(`research/*.py`)

- 只读 PostgreSQL,**绝不回写**事实表
- 复权口径与 `utils/adjusted_prices` 对齐(`raw_actions_v1`,后复权,因子含分红+拆股)
- 一致性测试锁定了 `utils.adjusted_prices.factor_for_date` 和 `research.data.apply_adjustment` 同口径,改任一边必须同步改另一边

### 提交信息

- 用 conventional commits 格式: `feat(area): ...` / `fix(area): ...` / `test(area): ...`
- 消息体说**为什么**改,不只说改了什么
- 不在 commit message 里加 "Generated by Codex" 之类的水印

### PR 描述模板

每个 PR 必须包含:

1. **任务来源**: 链接到 `docs/codex_tasks/<task-spec>.md`
2. **改动总览**: 新增/修改/删除文件,一行一个
3. **关键设计决策**: 走了哪条路、为什么(尤其是规格里留给实现者选的点)
4. **验收命令**: 任务规格"验收"小节里所有命令,贴 PR 时跑过的输出 tail
5. **风险**: 任何对其他模块的隐式假设、需要后续手动跑的命令(alembic / 数据回填等)
